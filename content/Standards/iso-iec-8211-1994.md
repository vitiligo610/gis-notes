# Philosophy of ISO 8211
ISO 8211 is the structural backbone of S-57 ENCs. It is self describing format that dictates how the data must be read. This entry explains the format of an ISO 8211 document so that one can look at raw binary content and understand the structure.

ISO 8211 was designed to be a self describing, linear, and platform independent format. The self describing here means that it tells the shape or schema of the data that it contains. Think of it as a single relational database. A database has a schema (`CREATE TABLE ...`) and data is added (`INSERT INTO ...`) following that schema.

You can read the document in a sequential order, the parser doesn't need to jump to multiple random locations to retrieve data. Since it is simply a binary format, there are no platform restrictions.

To understand this format clearly, you can use a hex editor to view ISO 8211 files or S57 files. You can use [ImHex](https://imhex.download/) or any hex editor of your choice. This [link](http://download.osgeo.org/gdal/data/s57/enctds/) contains examples of S-57 ENCs (files ending with `.000` extension).

> [!Tip]
> 
> If you are using ImHex editor, you can use this pattern file [iso8211.hexpat](https://gist.github.com/vitiligo610/f9e1962bc637909df7c29b2f9b43796d) to highlight different sections of logical records.
# Logical Records
Every ISO 8211 is built as collection of logical records. There are two types of logical records:
1. Data Descriptive Record (DDR)
2. Data Record (DR)

Every ISO 8211 document contains a single DDR at the top, and the rest of the file after that DDR consists of DRs.

![[Pasted image 20260628130820.png]]

The data descriptive record is responsible for laying out the structure of each data record following it. Consider it as a blueprint for the rest of the document. DDR is similar to the database schema we define before inserting data to tables.

Each record (both DDR and DR) contains 3 primary sections:
1. **Leader:** This is the preamble of the record.
2. **Directory Entry:** This is the allocation table. It specifies what fields are present in the field area section (the next section), what is the length of each field data and the offset
3. **Field Area:** Unlike the first two sections, field area section is different for DDR and DR. For DDR, field area includes descriptive fields (more on that below), whereas for DR it contains the raw payload (the actual data).

![[Pasted image 20260628123150.png]]
## Leader
The leader is the first thing that is included in every record. It is a fixed length 24-byte header that tells about the record it is part. If the leader of record isn't *exactly* 24 bytes, that file is either corrupted or doesn't follow the proper standard for ISO 8211 documents.

The contents included in that 24 byte below are shown in the table below:

| Relative Position (RP) | Length | Field Name                        | Typical Value        | Format                    | Remarks                                                             |
| ---------------------- | ------ | --------------------------------- | -------------------- | ------------------------- | ------------------------------------------------------------------- |
| 0 - 4                  | 5      | Record Length                     | e.g. `00128`         | ASCII digits, zero-padded | The total length of this record                                     |
| 5                      | 1      | Interchange Level                 | `3`                  | ASCII digit               |                                                                     |
| 6                      | 1      | Leader Identifier                 | `L` (DDR) / `D` (DR) | ASCII char                | Whether this leader is for DDR or DR                                |
| 7                      | 1      | In-Line Code Extension Indicator  | `E`                  | ASCII char                |                                                                     |
| 8                      | 1      | Version Number                    | `1`                  | ASCII digit               |                                                                     |
| 9                      | 1      | Application Indicator             | ` ` (space)          | ASCII char                |                                                                     |
| 10 - 11                | 2      | Field Control Length              | `09`                 | ASCII digits              | This length will be used in Field area section (more on that later) |
| 12 - 16                | 5      | Base Address of Field Area        | e.g. `00046`         | ASCII digits, zero-padded | The relative address from the start of the record.                  |
| 17 - 19                | 3      | Extended Character Set Indicator  | ` ! `                | ASCII chars               |                                                                     |
| 20                     | 1      | Entry Map: Size of Field Length   | e.g. `4`             | ASCII digit               | Will be used in the next section (directory entry)                  |
| 21                     | 1      | Entry Map: Size of Field Position | e.g. `5`             | ASCII digit               | Will be used in the next section (directory entry)                  |
| 22                     | 1      | Entry Map: Reserved               | `0`                  | ASCII digit (always 0)    |                                                                     |
| 23                     | 1      | Entry Map: Size of Field Tag      | e.g. `4`             | ASCII digit               | Will be used in the next section (directory entry)                  |

![[Pasted image 20260628124220.png]]
In the above image, the 24 red colored bytes are the leader of this DDR. We can see that it has record length of 1970, it is leader for DDR (because leader identifier is L), and the field area section starts at address `245` (base address). You can decode the rest of the information easily with the help of the table.
## Directory Entry
After exactly 24 bytes of the leader, directory entry section of the record starts. It is like an index which tells us what fields are included in the next section i.e. field area section. What are their tag names, what is the length of their payload/data in bytes. It also tells when the data for the field starts with respect to the base address extracted from the leader section.
![[Pasted image 20260628125510.png]]
So, directory entry section consists of entries that tell about what fields exist in the field area section. The exact length of tag, length and offset part is defined by the leader.

> [!Note]
> 
> The length of tag field, length field and offset field is mentioned in the leader section of the record. In leader section, the bytes 20 to 23 contain data related to entry map which contains these values.


![[Pasted image 20260628125837.png]]
Consider the hex dump of a data record as shown in the above image. The green part is the directory entry section for this record. There are a total of 3 entries in this record. A single entry is selected. The selected entry tells that the field area section (the blue section) contains a field with tag `0001`, the length of its data is `3` bytes and it is located at `00` offset from base address (from leader section) - meaning it is the first field in the section.

We can also see that after the green section ends, it has a byte of `0x1E`, this is called a field terminator. After all directory entries, a field terminator is added to indicate that the directory section has ended and now the next section i.e. field area section will begin.
## Field Area Section
As stated before, the difference between DDR and DR lies in their field area section. The field area sections of DDR and DR are different, but they are still related.
### Field Area Section of DDR
The field area of DDR defines the exact schema of every field used in the entire document. Earlier we talked about how a ISO 8211 document is a self describing document. It lays out the schema first and then inserts data according to the schema defined. The field area section of DDR *is* the schema.

This section consists of data descriptive fields. As the name suggests, they describe what the field is about and what subfields it contains. The following is the structure of a single data descriptive field.
![[Pasted image 20260628141810.png]]

> [!Note]
> UT means Unit Terminator, its value is `0x1F`. Whereas FT means Field Terminator, its value is `0x1E`.

A single field consists of single or multiple subfields. Each subfield has a name (in the array descriptor) and a format or data type (in format controls).

![[Pasted image 20260628134041.png]]
#### Field Controls
Field controls is a fixed length prefix of a data descriptive field. Its length is defined by the 10th and 11th byte of the leader section. In most cases, its length is almost always 9.

| Offset within Field Controls | Length | Name                            | Meaning                                                                                                                                                                                                                                                                                                                                                              |
| ---------------------------- | ------ | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 0                            | 1      | Data Structure Code             | `0` = elementary (single, non-repeating value)<br>`1` = vector / linear (one linear list of subfields)<br>`2` = array (repeating vectors like 2d table)<br>`3` = concatenated (mix of vector and array structure, multi dimensional)                                                                                                                                 |
| 1                            | 1      | Data Type Code                  | `0` = character string<br>`1` = implicit point (integer, decimal point position is fixed/implied)<br>`2` = explicit point (real number, decimal point is literally present in the text)<br>`3` = explicit point, scaled<br>`4` = character-mode bit string<br>`5` = bit string (binary)<br>`6` = mixed data types (this field's subfields are not all the same type) |
| 2 - 3                        | 2      | Auxiliary Controls              | Reserved for future use.                                                                                                                                                                                                                                                                                                                                             |
| 4                            | 1      | Printable Field Terminator      | Printable field terminator to use when writing programs to display ISO 8211 documents                                                                                                                                                                                                                                                                                |
| 5                            | 1      | Printable Unit Terminator       | Same idea, for the Unit Terminator.                                                                                                                                                                                                                                                                                                                                  |
| 6 - 8                        | 3      | Escape Sequence (Character Set) | An ISO 2022-style escape sequence identifying the character encoding used for `A`-type (character string) subfield data in this field. Commonly all spaces (default character set) in practice.                                                                                                                                                                      |
#### Field Name
It is the plain ASCII field name. It is of variable length and is terminated by unit terminator (UT).
#### Array Descriptors
Array descriptors consist of sub fields that are part of the field. A field can have a single or multiple sub fields. Array descriptor is different for different types of data structures:
- If the field's data structure is elementary that it consists of a single subfield. e.g. `DSID`
- If the field's data structure is vector, then it consists of multiple subfields joined by `!`. e.g. `DSNM!DSID|EDTN`
- If the field's data structure is array, then it consists of multiple subfields joined by `!`, and prefixed by `*`. The `*` indicates that the subfields following it are repeating. e.g. `*YCOO!XCOO`
- The concatenated data structure consists a mix of vector and array data structures. It is fairly complex and is not used by S57 standard, hence we will not focus on that in this document.
#### Format Controls
After array descriptors are terminated with UT, the format controls part of data descriptive field starts. It specifies the data types of the subfields mentioned in array descriptors. The format controls are enclosed in round brackets `()`. There are multiple types of data types as listed in table below:

| **Format Control** | **Data Type**          | **Meaning**                                                                                                                |
| ------------------ | ---------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **`A`**            | Character String       | Plain text string (ASCII or UTF-8).                                                                                        |
| **`I`**            | Implicit-Point Integer | A whole number stored as readable text characters (e.g., `"452"`).                                                         |
| **`R`**            | Explicit-Point Real    | A decimal number stored as readable text characters with a literal decimal point (e.g., `"12.34"`).                        |
| **`S`**            | Explicit-Point Scaled  | A number stored as a text string using scientific notation (e.g., `"1.23E+04"`).                                           |
| **`b`**            | Binary Data            | Raw binary bytes representing an integer or floating-point value instead of characters (e.g., `b14` for a 4-byte integer). |
| **`B`**            | Bit String             | A sequence of raw bits, usually evaluated as an array of binary flags or a bitmask.                                        |

The format controls can either be of fixed length or of variable length. If they are variable then we just write its code e.g. `A`. If it is of fixed width we enclose its byte width in the brackets after writing its code e.g. `A(6)`. This means that it is a plain ASCII string with length of 6 bytes.

The format controls map one to one with the subfield names in the array descriptors. Meaning, the number of sub field names, and number of format controls is exactly same. 

However, if multiple fields have same format, we can prevent the format to repeat multiple times and just write a multiplier before the repeating format. E.g. suppose the array descriptors are `YCOO,XCOO` and their format controls are `(b24, b24)`. We can also write the format control as `(2b24)` to save space.

### Field Area Section of DR
Once the data descriptive fields specify the format of fields and their subfields, the field area section of DR writes the raw data.

> [!Note]
> 
> It is not necessary for a single DR to have data for all the fields that are present in DDR. DR only writes data for fields it is concerned with.

You just read the data in the DR, figure out the fields it has and parse the raw data according to that field's format that was described earlier in DDR.

![[Pasted image 20260628142242.png]]

The blue section in the above image is the field area section of a data record. It doesn't contain any structural metadata, just raw data.
